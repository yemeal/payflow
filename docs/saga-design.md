## 1. TL;DR: ключевые решения одним экраном

| # | Решение | Обоснование | Статус |
|---|---------|-------------|--------|
| 1 | Оркестрация, не хореография | Явная машина состояний, проще отладка и добавление шагов, нет циклических зависимостей | Принято (ADR-001) |
| 2 | Состояние саги - в Postgres оркестратора, события - только триггеры | Состояние = источник правды; переживает рестарты; события могут дублироваться и опаздывать | Принято |
| 3 | Каждый переход состояния = одна транзакция БД: dedup + переход + команды в outbox | Атомарность "обработал событие ровно один раз и надёжно отправил команды" | Принято (ADR-002) |

| 6 | Изоляция через semantic lock (Order.status = PENDING) + guard по состоянию саги + SELECT FOR UPDATE | Сага даёт ACD без I; аномалии закрываются контрмерами, а не надеждой | Принято (ADR-004); отмена во время саги - бэклог |
| 7 | Структура саги: резерв с TTL (compensatable) -> оплата пользователем (pivot, дедлайн 30 мин) -> списание резерва и финализация (retriable) | Порядок шагов минимизирует компенсации: самое рискованное - перед pivot | Принято, флоу v2 в 9.8 |
| 8 | Ретраи шагов - у оркестратора (backoff + jitter, поле retry_after + фоновый поллер), транспортные ретраи - у участников | Не блокировать консюмер sleep'ом; разделение ответственности | Принято ТЗ, уточнено |
| 9 | Различать бизнес-отказ (payment declined -> компенсация сразу) и технический сбой (timeout -> retry) | Бизнес-отказ не лечится повтором и не должен попадать в DLQ | Принято: блок failure с обязательным retriable в каждом *.failed |
| 10 | Таймауты шагов: deadline_at у саги + поллер зависших саг; переотправка команды безопасна благодаря идемпотентности участников | Ответ может не прийти вовсе - оркестратор обязан замечать это сам | Принято; дедлайн - свойство шага (per-step, дефолт глобальный) |
| 11 | DLQ - только для poison-сообщений и исчерпанных технических ретраев; сага при этом -> FAILED + алерт | DLQ - не помойка и не путь для бизнес-отказов | Принято ТЗ |
| 12 | Финальность: любая сага обязана достигнуть терминального состояния (COMPLETED / CANCELLED / FAILED) | Заказ не может висеть вечно; FAILED = ручной разбор | Принято ТЗ |
| 13 | Единая таблица outbox во всех сервисах: + поля type (command/event), topic, key | Один переиспользуемый релей и одна механика везде | Принято (ADR-006) |

## 2. Теория: откуда растёт сага

### 2.1 Почему не распределённая ACID-транзакция

Классическое решение кросс-сервисной согласованности - двухфазный коммит (2PC/XA): координатор собирает у участников обещания (prepare), затем командует commit. Проблемы, из-за которых 2PC не используется в микросервисах:

- **Блокирующий протокол.** Между prepare и commit участники держат локи. Падение координатора оставляет участников заблокированными ("in-doubt" транзакции).
- **Доступность перемножается.** Транзакция успешна, только если доступны все участники одновременно; по CAP это выбор C в ущерб A на каждом шаге.
- **Стек не поддерживает.** Kafka, большинство NoSQL и облачных брокеров не участвуют в XA. Database-per-service делает единого координатора невозможным (AWS PG прямо пишет: "the two-phase commit is not an option").
- **Связность.** 2PC требует, чтобы участники доверяли чужому координатору управлять своими локальными транзакциями - это ломает автономию сервисов.

Pat Helland ("Life Beyond Distributed Transactions", CIDR 2007) формулирует вывод, который стал каноном индустрии: в системах "almost-infinite scale" атомарные транзакции существуют только внутри одной сущности (entity); между сущностями - только сообщения at-least-once, а значит бизнес-логика обязана строиться на идемпотентных операциях и явных "activities" со сверкой состояния. Сага - это ровно такая activity.

### 2.2 Оригинальная статья: Garcia-Molina, Salem, "Sagas" (SIGMOD 1987)

Сага придумана не для микросервисов, а для long-lived transactions (LLT) в одной БД: длинная транзакция держит локи часами и душит остальных. Ключевые идеи, дожившие до нас без изменений:

- **Определение.** LLT является сагой, если её можно записать как последовательность транзакций `T1..Tn`, которые допустимо чередовать с чужими транзакциями. Каждой `Ti` (кроме последней) сопоставлена компенсация `Ci`.
- **Гарантия.** Система обеспечивает один из двух исходов: либо `T1..Tn` выполнены полностью, либо выполнена префиксная часть и её компенсации: `T1..Tj, Cj-1..C1` (усечённый вариант отката).
- **"Последовательность двух транзакций - не транзакция".** Композиция транзакций не замкнута относительно атомарности - нужен протокол поверх.
- **Компенсация - семантический undo, а не rollback.** `Ci` отменяет действие `Ti` с точки зрения бизнеса, но не обязана вернуть байты БД в исходное состояние (пример: refund - это новая проводка, а не стирание старой).
- **Два вида восстановления.** Backward recovery - откат компенсациями. Forward recovery - сохранённые checkpoint'ы (saga log) и дожим саги вперёд до конца. На практике комбинируют: до "точки невозврата" - назад, после - только вперёд.

### 2.3 ACD вместо ACID

Ричардсон (Microservices Patterns, гл. 4) сжимает свойства саги в формулу: **Atomicity + Consistency + Durability, но без Isolation**. Каждая `Ti` - локально ACID, но промежуточные состояния саги видимы всем. Отсюда два системных следствия:

1. Откат надо программировать руками (компенсации - это бизнес-логика, а не механика СУБД).
2. Конкурентные саги порождают аномалии, и их надо давить контрмерами (раздел 3).

### 2.4 Структура саги: compensatable / pivot / retriable

Каноническая декомпозиция шагов (Richardson; подтверждена Azure Architecture Center):

| Тип шага | Определение | Пример в OrderFlow |
|---|---|---|
| **Compensatable** | Может быть отменён компенсацией; идёт до pivot | Резерв товара (компенсация: отмена резерва) |
| **Pivot** | Точка невозврата: после её успеха откат невозможен/не нужен; граница "go/no-go" | Списание денег |
| **Retriable** | Идёт после pivot; не может провалиться навсегда - ретраится до успеха; обязан быть идемпотентным | Финализация заказа (статус COMPLETED), уведомление |

Практическое правило проектирования: **переставляй шаги так, чтобы самые ненадёжные и самые "бизнес-отказные" шли раньше, а всё после pivot было тривиально ретраибельным**. Тогда компенсаций мало и они простые. Это же правило - контрмера pessimistic view (раздел 3.2).

### 2.5 Хореография vs оркестрация

| Критерий | Хореография (события, без центра) | Оркестрация (центральный координатор) |
|---|---|---|
| Связность | Низкая на вид, но участники знают о событиях друг друга -> риск циклических зависимостей | Участники знают только свои команды; всю топологию знает оркестратор |
| Видимость процесса | Размазана по сервисам; "куда делся заказ" - археология по логам | Состояние саги в одной таблице; тривиальный аудит |
| Эволюция (добавить шаг) | Трогаешь несколько сервисов | Трогаешь только оркестратор |
| Отказоустойчивость | Нет SPOF | Оркестратор - SPOF (лечится: stateless-процесс + состояние в БД + несколько реплик) |
| Сложность старта | Ниже для 2-3 шагов | Выше: отдельный сервис, машина состояний |
| Тестируемость | Интеграционный тест = поднять всех | Машину состояний можно юнит-тестить как чистую функцию |

Консенсус источников (Richardson, Azure, AWS): хореография - для простых, коротких саг; оркестрация - для всего, где шагов больше двух, есть компенсации и нужна наблюдаемость. Для OrderFlow выбрана оркестрация (ADR-001) - и это же выбор ТЗ.

Важное предостережение Ричардсона: оркестратор должен **оркестрировать, а не централизовать бизнес-логику**. Он решает "что дальше", но не знает, как резервировать товар. "Умные эндпоинты, глупый оркестратор" - иначе получится distributed monolith с богом-сервисом в центре.

## 3. Отсутствие изоляции: аномалии и контрмеры

Это главный источник продакшен-багов в сагах (и любимый вопрос на собеседованиях). Morling (InfoQ) оценивает уровень изоляции саг как "примерно read uncommitted".

### 3.1 Аномалии (на примерах OrderFlow)

| Аномалия | Суть | Наш пример |
|---|---|---|
| **Lost update** | Сага A и транзакция B читают одно, пишут по очереди, обновление A затёрто | Пользователь вызывает PATCH /orders/{id}/cancel, пока сага в PAYMENT_CHARGING; сага затем пишет COMPLETED поверх отмены |
| **Dirty read** | B читает промежуточное состояние саги A, которое потом откатится | Дашборд видит заказ "оплачивается", хотя через секунду резерв откатится |
| **Fuzzy / non-repeatable read** | Два чтения внутри одной саги видят разные данные | Оркестратор в начале саги посчитал сумму по items, а к моменту оплаты заказ изменён |

### 3.2 Шесть контрмер (Richardson, гл. 4.3; подтверждено Azure)

1. **Semantic lock** - флаг уровня приложения: запись в "неустойчивом" состоянии помечается `*_PENDING`, и другие транзакции либо ждут, либо отказывают быстро. Лок снимает финальная транзакция саги (успех или компенсация). **Это наш основной механизм**: `Order.status = PENDING` уже есть в скелете; правило - пока заказ PENDING, его нельзя менять мимо саги (отмена пользователем идёт командой в оркестратор, не прямым UPDATE).
2. **Commutative updates** - операции, дающие одинаковый результат в любом порядке (кредит/дебет счёта). Полезно для компенсаций: "cancel_reservation" после "reserve" должен быть корректен, даже если резерв уже истёк сам.
3. **Pessimistic view** - переупорядочить шаги саги так, чтобы грязные данные не были видны: то, что страшно откатывать, выполняй в retriable-фазе после pivot. У нас: заказ становится видимым как CONFIRMED/COMPLETED только после оплаты.
4. **Reread value** - перед записью перечитай значение и убедись, что оно не изменилось (optimistic offline lock); если изменилось - прерви/перезапусти сагу.
5. **Version file** - журналируй операции над записью, чтобы уметь их переупорядочивать/сверять (по сути event log на запись).
6. **By value** - выбирай механизм по цене риска: дешёвые операции - сагой, дорогие (деньги > X) - более строгими средствами (вплоть до 2PC там, где он возможен).

Для OrderFlow v1 берём: **semantic lock (1) + pessimistic view (3) + guard по состоянию саги** (событие, не соответствующее текущему состоянию, логируется и игнорируется) + **SELECT FOR UPDATE на строке саги** при каждом переходе (защита от конкурентной обработки двух событий одной саги, например при ребалансе консюмер-группы). Контрмеры 2 и 4 - точечно в участниках (идемпотентная отмена резерва; проверка версии заказа перед финализацией).

## 4. Надёжная доставка: outbox, идемпотентность, порядок

### 4.1 Dual write и transactional outbox

Антипаттерн: `commit в БД; producer.send(...)` - два разных хранилища без общей транзакции. Возможны оба вида рассинхрона (записали, но не отправили; отправили, но откатились). Решение (microservices.io, Morling):

- событие/команда пишется в таблицу `outbox_events` **в той же транзакции**, что и бизнес-изменение;
- отдельный relay-процесс публикует записи в брокер (polling publisher - наш случай - или CDC/Debezium);
- гарантия: сообщение уйдёт **тогда и только тогда**, когда транзакция закоммичена; доставка at-least-once.

В репозитории это уже реализовано (payment_service, перенесено в order_service). **Ключевой вывод отчёта: оркестратору outbox нужен так же, как и участникам** - команда "spиши деньги" не должна теряться, если Kafka моргнула после commit перехода (ADR-002). ТЗ предлагает компромисс "commit -> send + переотправка зависших при рестарте"; это рабочая forward-recovery схема, но outbox проще рассуждается и у нас уже есть готовая механика.

### 4.2 Идемпотентный консюмер

Kafka - at-least-once: ребаланс, падение до commit offset, ретраи продюсера дают дубликаты. Формула exactly-once processing: **at-least-once доставка + дедупликация на стороне потребителя**.

Каноническая реализация (Idempotent Consumer, microservices.io; journal table у Morling; наша processed_events):

```text
одна транзакция БД:
  INSERT INTO processed_events(event_id, ...) ON CONFLICT DO NOTHING
  если rowcount == 0 -> дубликат: выходим (после commit - ack offset)
  иначе -> бизнес-обработка (переход саги, запись команд в outbox)
commit
ack offset (после commit БД, не до)
```

Критично: проверка дубликата и бизнес-эффект - **в одной транзакции** (иначе race двух инстансов), и это атомарный INSERT .. ON CONFLICT, а не SELECT-затем-INSERT.

Участники (inventory, payment) дедуплицируют по `message_id` команды. Best practice для ответов: **сохранить результат первого выполнения и на дубликат команды переопубликовать тот же ответ** - оркестратор мог не получить ответ, и повторная команда - это его способ спросить "ну что там?".

### 4.3 Kafka-транзакции (EOS) - почему не полагаемся

Kafka EOS (idempotent producer + transactions + read_committed) даёт exactly-once только в цепочках consume-process-produce **внутри Kafka**. Как только в цикле появляется внешняя БД (наш случай - всегда), EOS её не покрывает. Поэтому индустриальный стандарт для сервисов с БД - outbox + идемпотентные консюмеры, а не Kafka-транзакции. (Conduktor/Confluent glossary; Java Code Geeks.)

### 4.4 Порядок сообщений

- Kafka гарантирует порядок только внутри партиции -> **все сообщения одной саги обязаны иметь один ключ партиционирования**. У нас: `key = order_id` (равноценно saga_id - соответствие 1:1). Это уже заложено в скелетах и в KafkaOutboxPublisher.
- Relay обязан публиковать outbox в порядке создания и останавливать батч на первом сбое (сохранение порядка) - уже реализовано в перенесённом OutboxRelayService.
- Несмотря на это, консюмер обязан переживать нарушение порядка (сеть, ретраи): guard по состоянию + идемпотентность закрывают "опоздавшие" события.

## 5. Машина состояний оркестратора

### 5.1 Принципы

1. **Состояние - источник правды, события - триггеры.** Решение "что делать" принимается по `(event_type, saga.status)`, никогда только по событию.
2. **Явный dispatcher** `{(event_type, status): handler}` вместо каскада if/elif (заложено ТЗ и скелетом). Неожиданная пара -> warning-лог + ack (дедуп-запись всё равно фиксируем), не исключение.
3. **Каждый переход - транзакция**: dedup + загрузка саги FOR UPDATE + мутация + команды в outbox + commit. Никаких побочных эффектов до commit.
4. **Терминальные состояния абсорбирующие**: из COMPLETED / CANCELLED / FAILED переходов нет; события к ним - log + ignore.
5. **Переходы - чистые функции** `(saga, event) -> (new_state, [commands])`; вся грязь (БД, Kafka) - вокруг. Это делает машину тривиально юнит-тестируемой.

### 5.2 Персистентность и конкурентность

- `SELECT ... FOR UPDATE` на строке саги - два события одной саги обрабатываются строго последовательно даже при нескольких инстансах оркестратора.
- Альтернатива/дополнение - optimistic locking (колонка `version`, как у Morling). При FOR UPDATE строго необходимой нет; полезна как страховка от "потерянного" UPDATE мимо блокировки.
- Партиционирование по order_id само по себе сериализует события одной саги в одном консюмере - но полагаться только на это нельзя (ретраи, DLQ-redrive, ручные операции ломают предпосылку).

### 5.3 Команды vs события

| | Команда | Событие |
|---|---|---|
| Семантика | Императив конкретному адресату ("зарезервируй") | Свершившийся факт ("зарезервировано") |
| Топик | Топик адресата: `inventory.commands`, `payments.commands` | Топик источника/шины: `orders.events`, `payments.events` |
| Кто шлёт | Оркестратор | Участники и order_service |
| Идентификатор | `command_id` (= message_id для дедупа участника) | `event_id` (для дедупа оркестратора) |

Сквозная корреляция: `saga_id` + `order_id` в каждом сообщении; желательно также `causation_id` (id сообщения-причины) - дёшево, а отладка распределённого потока становится линейной.

Контракты - contract-first (ТЗ N3): metadata (`command_id/event_id`, тип, `version`, timestamp, source) + data. Эволюция - только аддитивная в рамках major-версии; консюмеры - tolerant reader (лишние поля игнорируются - у нас уже так через Pydantic).

## 6. Отказы: ретраи, таймауты, DLQ, компенсации

### 6.1 Классификация ошибок - до того, как ретраить

| Класс | Примеры | Реакция |
|---|---|---|
| **Транзиентная техническая** | timeout, 5xx, брокер недоступен, deadlock | Retry с backoff + jitter |
| **Постоянная техническая (poison)** | Невалидный формат события, неизвестный тип, ошибка кода | Немедленно в DLQ + сага FAILED (ретраи бессмысленны) |
| **Бизнес-отказ** | Недостаточно средств, товара нет, карта отклонена | НЕ ретраить и НЕ в DLQ: это легитимный исход -> компенсация + CANCELLED |

ТЗ упрощает: `*.failed` от заглушек трактуется как транзиентный сбой и ретраится. Для боевой интеграции с payment_service событ'`payment.failed` должно нести признак (`failure.retriable: bool` или код), иначе оркестратор будет "долбить" отклонённую карту. Заложить поле в контракт сейчас - дёшево.

### 6.2 Ретраи

- **Exponential backoff + jitter обязателен** (2s, 4s, 8s, +-20%): без jitter ретраи синхронизируются в "волны" (thundering herd).
- **Не спать в консюмере.** `asyncio.sleep` в обработчике останавливает партицию. Схема ТЗ правильная: `retry_count`, `retry_after` в saga_state; отдельный фоновый поллер раз в ~секунду переотправляет команды созревших саг (через outbox - просто кладёт команду повторно с тем же command_id).
- retry_count сбрасывается при переходе на следующий шаг; лимит (3) - на шаг, не на сагу.
- Транспортные ретраи (подключение к брокеру, HTTP к провайдеру) - забота участников (у payment_service уже есть tenacity + circuit breaker); оркестратор ретраит только шаги саги.

### 6.3 Таймауты шагов

Ответ может не прийти вовсе (участник умер после дедупа, но до публикации ответа). Оркестратор обязан обнаруживать это сам:

- при отправке команды проставляем `deadline_at = now + step_timeout`;
- тот же фоновый поллер выбирает саги с `deadline_at < now` в нетерминальных состояниях;
- реакция = как на транзиентный отказ: retry_count++, переотправка команды (безопасно: участник идемпотентен и переопубликует прежний ответ), после лимита - компенсация/FAILED.

Это forward recovery из оригинальной статьи, применённый к "зависшим" сагам; тот же механизм закрывает и рестарт оркестратора с сагами в промежуточных состояниях.

### 6.4 Dead Letter Queue

- Попадает: poison-сообщения (сразу) и шаги после исчерпания технических ретраев.
- Формат: оригинальное сообщение + метаданные (saga_id, retry_count, last_error, failed_at, source topic/partition/offset) - иначе разбор невозможен.
- DLQ-reader: отдельная consumer group, ERROR-лог (в проде - алертинг). Сага при этом переводится в FAILED - **DLQ не заменяет финализацию саги**.
- Re-drive (ручная переотправка после починки) должен быть идемпотентным по построению - у нас так, потому что дедуп по event_id/command_id никуда не девается.

### 6.5 Компенсации

Правила (Sagas 1987; McCaffrey; Temporal; Azure Compensating Transaction):

1. Компенсируются только уже выполненные шаги, **в обратном порядке**.
2. Компенсация - retriable-операция: она **не имеет права провалиться навсегда**; её нельзя "отменить" (не бывает компенсации компенсации). После N неудач - FAILED + ручной разбор (наш выбор, совпадает с ТЗ).
3. Компенсация идемпотентна и коммутативна со "своим" действием: `cancel_reservation` корректен, даже если резерв не существует / уже истёк / уже отменён (ответ "already cancelled" - это успех).
4. Компенсация - бизнес-действие с последствиями (refund идёт через банк днями), а не DELETE строки.
5. Шаги после pivot не компенсируются - только дожимаются вперёд.
6. Полукомпенсируемые эффекты (отправленный email) компенсируются "письмом-извинением" (Helland: "I'm sorry"-логика) либо откладываются в retriable-фазу.

### 6.6 Инвариант финальности

Любая сага достигает терминального состояния за конечное время: успех -> COMPLETED; бизнес-отказ или успешная компенсация -> CANCELLED; исчерпание ретраев (включая компенсацию) -> FAILED + алерт. "Вечных" промежуточных состояний не существует по построению - это гарантируют таймаут-поллер и лимиты ретраев. FAILED - это сигнал тревоги (несколько в сутки = системная проблема), а не ветка нормальной логики.

## 7. Наблюдаемость и эксплуатация

- **Логи**: каждый переход - структурированная запись с saga_id, order_id, event_id/command_id, from_state -> to_state, retry_count. По saga_id должен восстанавливаться полный путь (критерий ТЗ N2).
- **Метрики**: счётчики саг по статусам; гистограмма времени до терминального состояния; счётчик компенсаций; глубина DLQ; лаг консюмер-групп; глубина outbox (PENDING старше X - алерт: relay умер).
- **Алерты**: FAILED > 0; саги в нетерминальном состоянии дольше SLA; DLQ непустой; consumer lag растёт.
- **Админ-операции** (минимум, можно CLI/SQL): список зависших саг; ручной retry шага; ручной abort (запуск компенсаций); re-drive DLQ. Morling: "unprocessed messages block saga progression indefinitely" - за outbox и DLQ надо следить, они не самоочищаются.
- **Трейсинг** (позже): пробрасывать trace context в metadata сообщений; span на шаг саги.

## 8. Альтернативы саге (и почему всё-таки сага)

| Подход | Суть | Почему нет (у нас) |
|---|---|---|
| **2PC / XA** | Синхронный атомарный коммит у всех участников | Kafka/стек не поддерживает; блокирующий; убивает автономию и доступность |
| **TCC (Try-Confirm-Cancel)** | Участники резервируют ресурс (Try), затем Confirm/Cancel; координация на клиенте | По сути сага с явной фазой резервирования; наш резерв склада - это и есть Reservation pattern внутри саги |
| **Event sourcing всей системы** | Состояние = лог событий; outbox не нужен | Радикально другая модель хранения; учебная цель - классические таблицы + сага |
| **Workflow-движок (Temporal, Camunda, AWS Step Functions)** | Durable execution: код воркфлоу переживает падения, ретраи/таймеры/история из коробки | Индустриально это сильный выбор (Temporal: "retries failed activities automatically... resumes exactly where it left off"). Для OrderFlow отвергнут осознанно: цель - построить машину состояний руками и понять механику. Критерии пересмотра - в ADR-003 |
| **Ничего (монолитная транзакция)** | Один сервис, одна БД | Ломает database-per-service; но помнить: **лучшая сага - та, которой нет**; не дробить транзакцию без нужды |

У движков стоит подсмотреть три идеи, даже делая своё: durable timers (наш deadline-поллер - их бедный родственник), полная история саги (мы приближаем это логом переходов + processed_events), описание воркфлоу кодом с автоматической регистрацией компенсаций (Temporal Saga API: push компенсации в стек при успехе шага, unwind при сбое).

### 9.1 Определение саги заказа

| Шаг | Команда -> Участник | Успех (событие) | Отказ (событие) | Тип шага | Компенсация |
|---|---|---|---|---|---|
| 1. Резерв товара | `reserve_inventory` -> inventory | `inventory.reserved` | `inventory.reserve-failed` | Compensatable | `cancel_reservation` -> `inventory.reservation-cancelled` |

INVENTORY_RESERVING -> CANCELLED`; отказ оплаты `... PAYMENT_CHARGING -> COMPENSATING_INVENTORY -> CANCELLED`; невосстановимое `-> FAILED`.

### 9.2 Данные оркестратора (доработка скелета)

```sql
-- sagas: + поля таймаута/диагностики (skeleton: id, order_id, status, payload, retry_count, retry_after)
-- итерация 2 (ADR-006): целевая схема generic - плюс saga_type, business_key
--   c UNIQUE(saga_type, business_key) и current_step; order_id растворяется в business_key
ALTER TABLE sagas ADD COLUMN deadline_at timestamp NULL;   -- таймаут текущего шага
ALTER TABLE sagas ADD COLUMN last_error   text      NULL;  -- диагностика последнего сбоя
-- (опционально) version int NOT NULL DEFAULT 0 - optimistic lock поверх FOR UPDATE

-- processed_events: уже в скелете (event_id PK, saga_id, event_type, processed_at)

-- outbox оркестратора: как outbox_events order_service + адресация,
-- потому что у оркестратора адресатов несколько
CREATE TABLE outbox_messages (
  id         uuid PRIMARY KEY,            -- = command_id/event_id (дедуп у получателя)
  topic      text NOT NULL,               -- inventory.commands / payments.commands / orders.events
  key        text NOT NULL,               -- order_id: партиционирование = порядок саги
  message_type text NOT NULL,             -- reserve_inventory / saga.completed / ...
  payload    jsonb NOT NULL,
  status     outbox_status NOT NULL DEFAULT 'PENDING',
  attempts   int NOT NULL DEFAULT 0,
  last_error text NULL,
  created_at timestamp NOT NULL, updated_at timestamp NULL
);
```

### 9.3 Обработка события (сердце оркестратора, псевдокод)

```python
async def on_event(envelope: EventEnvelope) -> None:
    async with uow:                                   # ОДНА транзакция БД
        fresh = await processed_events.try_mark(envelope.event_id, ...)
        if not fresh:
            log.info("duplicate_event_skipped", event_id=...)
            return                                     # commit дедуп-noop -> ack

        saga = await sagas.get_for_update(envelope.saga_id)  # SELECT FOR UPDATE
        handler = TRANSITIONS.get((envelope.event_type, saga.status))
        if handler is None:                            # опоздавшее/чужое событие
            log.warning("unexpected_event_ignored", ...)
            return                                     # дедуп зафиксирован -> ack

        result = handler(saga, envelope.data)          # чистая функция:
        saga.status = result.new_status                #   (saga, event) ->
        saga.retry_count = result.retry_count          #   (new_status, [messages])
        saga.deadline_at = result.deadline_at
        for msg in result.messages:                    # команды и saga.*-события
            await outbox.add(msg)                      # в ту же транзакцию
        await sagas.update(saga)
    # commit прошёл -> FastStream ack'ает offset; relay опубликует messages
```

Фоновые циклы (оба - трюк одного поллера, интервал ~1s):

```python
# retry: саги с retry_after <= now -> переотправить команду шага (тот же command_id)
# timeout: саги с deadline_at <= now в нетерминальном статусе ->
#     retry_count += 1; if retry_count > MAX: компенсация или FAILED (+DLQ)
#     else: переотправка команды, deadline_at = now + step_timeout * backoff(retry_count)
```

### 9.6 Тестирование саги

- **Юнит**: TRANSITIONS как чистые функции - таблица (state, event) -> (state', commands); все "неожиданные" пары дают ignore.
- **Интеграционные** (testcontainers, как в payment_service): happy path; отказ inventory; отказ payment + компенсация; дубликат каждого события (стресс ТЗ: 100x order.created -> одна сага); рестарт оркестратора посреди саги; таймаут шага (участник молчит); компенсация падает N раз -> FAILED + DLQ.
- **Хаос-минимум**: kill -9 оркестратора между commit БД и ack offset (должен переварить дубль); остановка Kafka при commit перехода (outbox дошлёт).

### 9.7 Generic-ядро саги (итерация 2, ADR-006)

Слои оркестратора:

```text
SagaDefinition (данные, не код)      исполнитель (generic)            порты
  order-fulfillment:                  load saga FOR UPDATE             SagaRepository
    steps:                            dedup(event_id)                  ProcessedEvents
      reserve  -> inventory.commands  guard (event, step, status)      Outbox
        compensation: cancel          apply step outcome               
        timeout: 30s                  emit commands/events -> outbox   
      charge   -> payments.commands   advance / compensate / fail      
        pivot: true                   
        timeout: 30m (ожидание оплаты)
      commit_reservation -> inventory.commands   # retriable, после pivot
```

Правила разделения ответственности: оркестратор владеет процессом (порядок, ретраи, компенсации, таймауты) и не знает бизнес-правил шагов; участники владеют операциями (как резервировать, как списывать) и не знают о процессе. Определение саги - данные: его можно валидировать, версионировать и юнит-тестить отдельно от исполнителя.

Корреляция без saga_id у участников: команда несёт `metadata: {command_id, saga_id, business_key, ...}`; участник обрабатывает `data`, а metadata возвращает в ответе как есть (echo, EIP Correlation Identifier + Return Address). Дедуп у участника - по command_id.

### 9.8 Флоу заказа v2: резерв с TTL и оплата пользователем

Решение владельца: резерв живёт 30 минут; пользователь предупреждается и уходит на оплату; нет оплаты за 30 минут - резерв снимается и заказ отменяется; оплата пришла - в одной транзакции inventory списывает товар и снимает резерв.

Следствия для саги (это уже не "мгновенная" сага, а long-running с human-in-the-loop):

- Шаг оплаты превращается в "создать платёж и ждать исхода" с per-step дедлайном ~30 минут. Наш deadline-поллер это уже покрывает по конструкции: PAYMENT_WAITING с deadline_at = now + 30m; таймаут = бизнес-отказ (не ретраить) -> компенсация резерва -> CANCELLED.
- Появляется третья команда inventory: `commit_reservation` (списание + снятие резерва атомарно, retriable-шаг после pivot).
- Инвариант конфигурации: TTL резерва >= дедлайн оплаты + буфер (например 30m + 5m), иначе легальная оплата может прийти к истёкшему резерву.
- Гонка "оплата успела, резерв истёк" всё равно возможна (буфер - вероятностная защита). Решение итерации 3 - вариант (в): гонка считается нарушением инварианта конфигурации; commit_reservation на истёкшем резерве -> сага в FAILED на ручной разбор + метрика на каждое срабатывание.
- Дедлайны per-step: свойство шага в SagaDefinition, глобальный дефолт в Settings. Ориентир: p99 участника x 2-3 (inventory - секунды, платёж с участием человека - десятки минут). Единый таймаут на все шаги не работает: он либо душит быстрые шаги, либо ложно убивает медленные.

Цена и каталог: order_service ничего не знает о товарах и ценах - он владеет только флоу заказа. Решение итерации 3: валидация суммы - синхронная и до саги, прайсинг-шага в саге нет. Две точки: прогрев на UI до кнопки "подтвердить оформление" (пользователь видит провалидированную сумму заранее) и авторитетный пересчёт каталогом при POST /orders - сумме из тела запроса не доверяем, тот же принцип, что с user_id из JWT. Цена в заказе - снапшот на момент оформления. До появления каталога total_amount от клиента - зафиксированное упрощение MVP.

| GET | /admin/v1/sagas/{saga_id} | Сага + payload + история переходов |
| GET | /admin/v1/sagas/stuck?older_than= | Нетерминальные саги старше порога |

Мутации (ручной retry, abort, re-drive) - бэклог; в MVP API строго read-only. Для истории переходов - append-only таблица `saga_transitions(id, saga_id, from_status, to_status, event_id, command_ids, created_at)`: пишется в ту же транзакцию перехода, делает отладку и аудит тривиальными.

Метрики - Prometheus (закладываем сразу): `/metrics` у HTTP-сервисов (prometheus-client); у Kafka-only процессов - мини-HTTP экспортер в том же процессе. Базовый набор: sagas_total{saga_type,status}, saga_duration_seconds{saga_type}, saga_compensations_total, outbox_pending{service}, dlq_messages_total{topic}, consumer_lag. Prometheus-контейнер и дашборды - бэклог.

### 9.10 DLQ: конвенция, watcher, re-drive

- **Конвенция топиков**: у каждого рабочего топика есть парный `<topic>.dlq` (orders.events.dlq, inventory.commands.dlq, payments.commands.dlq, ...). Пишет в DLQ тот консюмер, который не смог обработать: poison - сразу, транзиентная ошибка - после исчерпания ретраев.
- **Poison-сообщение** - то, что детерминированно не может быть обработано: невалидный JSON, не проходит схему, неизвестный тип, нарушенный инвариант. Ретраить бессмысленно (результат тот же), а без DLQ оно "отравляет" партицию бесконечным nack-циклом. Важно: бизнес-отказ (карта отклонена) - НЕ poison и вообще не для DLQ, это легитимный исход шага.
- **Конверт DLQ** (фиксируется в contracts/): `original` (сообщение как есть) + `dlq_meta: {source_topic, partition, offset, consumer_group, error_class, error_message, retry_count, redrive_count, failed_at}`.
- **dlq_watcher - отдельный сервис**: подписка по паттерну `.*\.dlq$` (Kafka поддерживает regex-подписку), структурный ERROR-лог каждого сообщения + мок-алерта (лог "ALERT ..." вместо PagerDuty) + инкремент dlq_messages_total. Никакой логики обработки - только видимость.
- **Re-drive (переигровка)**: republish `original` в `source_topic` с тем же message_id, `redrive_count + 1`. Безопасность даёт транзакционность обработки: раз сообщение в DLQ - его эффект и dedup-запись НЕ закоммичены, значит повторная обработка чиста; если эффект всё же был - дедуп по message_id отбросит дубль. Правила: re-drive только после устранения причины (деплой фикса); лимит redrive_count (2), дальше - только ручной разбор; retention DLQ-топиков длиннее обычного. Инструмент (CLI или admin API) - бэклог.

### 9.11 Контракты: структура contracts/ (ADR-007)

```text
contracts/                        # "виртуальный отдельный репозиторий"
├── README.md                     # конвенции: именование топиков, эволюция, правила echo
├── CHANGELOG.md                  # semver по каждой схеме
├── envelope/
│   ├── command-metadata.schema.json   # command_id, saga_id, business_key, type, version, timestamp, source
│   ├── event-metadata.schema.json     # event_id, correlation (echo), type, version, timestamp, source
│   └── dlq-envelope.schema.json
├── orders/order-created.v1.schema.json
├── inventory/{reserve-command,reserved,reserve-failed,...}.v1.schema.json
└── payments/{charge-command,succeeded,failed,...}.v1.schema.json
```

- Для всех `*.failed`: блок `failure: {code, message, retriable}` - **обязателен** (retriable решает: ретрай или немедленная компенсация).
- Правило идемпотентности участника (README контрактов): журнал command_id -> результат; на дубль команды действие не выполняется, переиздаётся сохранённый ответ.
- Сервисы НЕ импортируют contracts/ в рантайме: у каждого свои Pydantic-модели, а contract-тест сервиса валидирует примеры/фикстуры против JSON Schema (осознанная дупликация - цена автономии).
- Эволюция: аддитивные изменения - minor; breaking - major + новая схема `*.v2.schema.json` (при необходимости - новый топик с суффиксом версии). Consumer = tolerant reader.

### 9.12 Notification service (задел)

Для отработки полукомпенсируемых действий: notification_service (мок email) с топиком `notifications.commands` (send_email) и парным .dlq. Место в саге - retriable-шаги после pivot ("заказ оплачен") и компенсационные уведомления ("заказ отменён" - Helland-овское "I'm sorry"-письмо вместо невозможного отзыва уже отправленного). Уведомления не влияют на исход саги: их отказ не триггерит компенсаций бизнес-шагов. Реализация - бэклог; в контрактах место зарезервировано.

### ADR-002: Доставка команд оркестратора

**Status:** Accepted (2026-07-14, владелец: вариант B, "без ресенд-поллера, по-умному через аутбокс и реле")

| Dimension | A: publish после commit + resend-поллер (как в ТЗ) | B: transactional outbox + relay (рекомендуется) |
|---|---|---|
| Сложность | Ниже (нет relay) | Выше на процесс, но механика уже готова в репо |
| Гарантии | Команда может потеряться до следующего скана зависших | Команда уйдёт тогда и только тогда, когда переход закоммичен |
| Задержка | Меньше на ~poll interval | +poll interval relay (у нас 2s) |
| Однородность | Особый путь для оркестратора | Один паттерн во всех сервисах (payment, order, orchestrator) |

**Решение:** B. Timeout-поллер (9.3) остаётся - но как страховка от молчания участников, а не как механизм доставки.

### ADR-003: Состояние саги - своя FSM в Postgres vs workflow-движок

**Status:** Accepted

| Dimension | Своя FSM (Postgres) | Temporal/Camunda/Step Functions |
|---|---|---|
| Учебная ценность | Максимальная (цель проекта) | Низкая (механика скрыта) |
| Надёжность из коробки | Строим сами (outbox, дедуп, таймеры) | Durable execution, ретраи, история |
| Операционная цена | Наши таблицы и поллеры | +кластер движка / вендор |
| Гибкость контрактов | Полная (Kafka-first) | Движок диктует модель |

**Решение:** своя FSM. **Критерии пересмотра:** саг становится > 2-3 типов; появляются длинные (дни) процессы с таймерами; человеко-шаги (approve); команда устала чинить самописные таймеры - тогда Temporal.

### ADR-004: Изоляция

**Status:** Accepted (2026-07-14: "изоляция по-максимуму")
**Решение:** semantic lock (Order.status=PENDING, изменения только через сагу) + pessimistic view (оплата - pivot, финализация - retriable) + guard `(event, state)` + SELECT FOR UPDATE на строке саги. Optimistic version - опционально позже. **Последствия:** пользовательская отмена заказа во время активной саги в MVP отвечает 409 (fail fast); полноценная отмена через команду cancel_saga - в бэклоге (13).

### ADR-005: Политика ретраев и DLQ

**Status:** Accepted (2026-07-14, подтверждено владельцем)
**Решение:** шаговые ретраи у оркестратора (3 попытки, exp backoff 2/4/8s + jitter 20%, retry_after + поллер); транспортные ретраи и circuit breaker - у участников; DLQ только для poison и исчерпанных технических отказов; бизнес-отказ -> немедленная компенсация без ретраев (блок failure с обязательным retriable в *.failed - зафиксировано в контрактах, 9.11).

### ADR-006: Generic-ядро саги и единый outbox

**Status:** Accepted (2026-07-14)
**Context:** сага понадобится не только для оформления заказа; добавление нового бизнес-процесса не должно требовать миграций сущностей участников и правок их кода.

| Dimension | A: сага "заказа" зашита в модели (saga_id в orders и т.д.) | B: generic-ядро + декларативные определения (принято) |
|---|---|---|
| Добавление новой саги | Миграции у участников, правки моделей | Новое определение в оркестраторе + контракты |
| Связность | Участники знают о процессе | Участники process-agnostic |
| Сложность ядра | Ниже | Выше на один уровень абстракции |
| Переиспользование кода | Слабое | Ядро, outbox, relay, dedup - общие |

**Решение:**
- Единая таблица `outbox` во всех сервисах: `id (=message_id), type (command | event), topic, key, message_type, payload, status, attempts, last_error, created_at`. Роутинг делает `topic`; `type` - семантика для метрик и отладки. Релей generic и одинаков везде.
- Ядро оркестратора generic: `sagas(id, saga_type, business_key, status, current_step, payload, retry_count, retry_after, deadline_at, last_error, created_at, updated_at)` + `UNIQUE(saga_type, business_key)` - идемпотентное создание саги по дубликату стартового события.
- Конкретные саги - декларативные определения (SagaDefinition): упорядоченные шаги, для каждого - команда, топик, компенсация, per-step timeout, маппинг событий на исходы. Машина исполнения одна на все определения.
- Участники не хранят и не интерпретируют saga_id. Корреляция: business key (для заказа - order_id) + правило echo: участник обязан вернуть в ответном событии correlation-метаданные команды (saga_id, command_id) как непрозрачные значения. В бизнес-таблицах участников их нет; журнал дедупликации - не бизнес-таблица.
- Order без saga_id: сагу создаёт оркестратор по order.created, saga_id генерирует сам.

**Consequences:** (+) новые саги дёшевы, код ядра один; (-) определение саги - ещё один DSL-слой, его надо тестировать как данные; (-) сквозной поиск "по заказу" в оркестраторе идёт через business_key, а не FK.

### ADR-007: Где живут контракты сообщений

**Status:** Accepted (2026-07-14, направление)
**Context:** легенда проекта - "это не монорепозиторий": сервисы принадлежат разным командам, общий импортируемый пакет запрещён.

| Опция | Суть | Оценка |
|---|---|---|
| A: только README | Формат описан текстом в документации | Не проверяемо машиной, дрейфует от кода |
| B: shared-пакет с моделями | Общий импорт для всех сервисов | Ломает легенду мультирепо; связывает релизы команд |
| C: contracts/ как "виртуальный отдельный репозиторий" (принято) | JSON Schema + версии + changelog; каждый сервис держит СВОИ Pydantic-модели и сверяет их с эталоном contract-тестами | Как в реальности: спека - отдельный артефакт, реализация у каждой команды своя |
| D: Schema Registry (Confluent) | Схемы регистрируются в рантайме, брокер-уровневая проверка совместимости | Правильная следующая ступень; для MVP - лишняя инфраструктура (бэклог) |

**Решение:** C. В реальных проектах это же делается так: отдельный git-репозиторий контрактов или registry (Avro/Protobuf/JSON Schema), плюс спецификации AsyncAPI как документация шины и consumer-driven contract-тесты (Pact) в CI. README-only - худший вариант, дупликация моделей per-service - не баг, а честная цена автономии команд. Детали структуры - 9.11.

## 11. Анти-паттерны (чего не делаем)

1. `producer.send` до/внутри транзакции БД (dual write) - только outbox.
2. `asyncio.sleep` в обработчике консюмера как "ретрай" - блокирует партицию.
3. Auto-commit offset / ack до commit БД - потеря событий при падении.
4. SELECT-затем-INSERT для дедупа - race; только атомарный ON CONFLICT.
5. Ретраить бизнес-отказы; складывать их в DLQ.
6. Компенсация pivot-шага или "компенсация компенсации".
7. Оркестратор лезет в чужие БД (обновить orders напрямую) - только события/команды.
8. Бизнес-логика участников в оркестраторе (бог-сервис).
9. События без event_id/saga_id/version - невозможно ни дедуплицировать, ни коррелировать, ни эволюционировать.
10. Разные ключи партиционирования для сообщений одной саги - потеря порядка.
11. Логировать полный payload с PII/токенами (у нас уже правило: токены - первые 8 символов).
12. Считать DLQ терминальным состоянием саги - сага при DLQ обязана перейти в FAILED явно.

## 12. Итерация 2: решено / открыто

**Решено владельцем 2026-07-14:**

1. Отмена заказа во время саги: MVP - fail fast (409); команда cancel_saga - бэклог.
2. TTL резерва: автоистекающий, 30 минут; пользователь предупреждается и оплачивает в этот срок; оплата -> inventory атомарно списывает товар и снимает резерв (commit_reservation); нет оплаты -> резерв снят, заказ отменён. Дизайн - 9.8.
3. Multi-item: v1 - резерв заказа атомарный ("всё или ничего"); частичные резервы не делаем.
4. Источник цены - только каталог (источник истины); order_service о товарах не знает, он владеет исключительно флоу заказа. До появления каталога total_amount от клиента - зафиксированное упрощение MVP.
5. Параллельные шаги - бэклог (понадобятся с notification_service).
6. Метрики - Prometheus, закладываем сразу (9.9).
7. retriable в *.failed - обязательный блок failure (9.11).
8. Дедлайны - per-step в SagaDefinition с глобальным дефолтом (9.8).
9. Контракты - contracts/ как виртуальный отдельный репозиторий (ADR-007).
10. DLQ - парный .dlq у каждого топика + отдельный dlq_watcher (9.10).

**Решено итерацией 3 (2026-07-14):**

1. **Гонка "оплата успела, резерв истёк"**: вариант (в) - гонка исключается инвариантом конфигурации (TTL резерва >= дедлайн оплаты + буфер, например 35m >= 30m + 5m); срабатывание трактуется как нарушение инварианта -> сага в FAILED на ручной разбор + метрика на каждый случай.
2. **Прайсинг**: синхронно и до саги. Пользователь видит провалидированную каталогом сумму ещё до нажатия "подтвердить оформление"; при POST /orders order_service авторитетно пересчитывает сумму через каталог (сумме из тела не доверяем - тот же принцип, что user_id из JWT). Прайсинг-шага в саге нет.
3. **Контракт оплаты**: участник шага оплаты - реальный payment_service (уже реализован, с моком провайдера). Оркестратор шлёт command_type=payment.process в payments.commands (формат CommandMetadata payment_service), исходы читает из payments.events (payment.completed / payment.failed); единственная правка payment_service - обязательный блок failure.retriable в payment.failed (чеклист 9.5).
4. **Payload саги**: минимальный снапшот, нужный командам и компенсациям: order_id, user_id, items[{product_id, quantity, price}], total_amount, currency. Без email, токенов и платёжных данных (email для уведомлений notification_service резолвит по user_id сам). Retention терминальных саг: пока не удаляем; позже - скрипты по created_at или помесячное партиционирование.
5. **Идемпотентность ответов участников**: через сервис идемпотентности участника - журнал command_id -> сохранённый результат; на дубль команды действие не выполняется, переиздаётся тот же ответ. Правило фиксируется в contracts/README (у payment_service механика уже есть - IdempotencyService).
6. **processed_events / outbox(SUCCESS)**: пока не удаляем; позже - те же скрипты по created_at.

**Решено итерацией 4 (2026-07-14, в ходе имплементации):**

1. **Корреляция - транспортная метадата, а не домен.** payment_service (и любой участник) НЕ хранит sagaId/businessKey в своих бизнес-сущностях: домену платежа безразлично, кто его коррелирует. Но echo для отложенных исходов (payment.completed рождается позже, в reconciliation-цикле, где контекста сообщения уже нет) требует, чтобы корреляция пережила время. Поэтому она живёт в отдельном инфраструктурном журнале сообщений `command_correlations` (command_id = idempotency_key платежа -> JSONB correlation): пишет её messaging-entrypoint ДО создания платежа, подставляет в конверт транспортный адаптер (CorrelationEnrichingPublisher). Domain/application слои о correlation не знают вообще. Тот же принцип - для inventory.
2. **customerId в платеже - это пользователь, а не заказ.** Единственный канал связи события платежа с сагой - `metadata.correlation` (echo). Никаких "запихнём orderId в customerId".
3. **Участник подстраивается под контракт, а не контракт под участника**: payment_service выровнен по общему конверту (echo + failure.retriable), а не generic-ядро под его особенности. Иначе каждый новый участник тянул бы свой канал корреляции в ядро.
4. **Именование полей единого outbox**: `kind` (COMMAND | EVENT) + `type` (значение из конверта: "inventory.reserve", "saga.completed"). Прежнее `type` + `message_type` читалось как одно и то же.
5. **TRANSITIONS-словарь уровня модуля отменён** (то же глобальное состояние, вид сбоку): определения саг - immutable-данные (SagaDefinition), собираются фабрикой `create_saga_registry(settings)` и раздаются через DI как SagaRegistry (APP scope). Диспетчеризация - методы реестра. Fail-fast на старте: реестр валидирует определения (уникальность шагов, единственный pivot, отсутствие компенсаций после pivot, инвариант TTL >= дедлайн оплаты + буфер).

**Открытых вопросов нет.** Идёт имплементация MVP по чеклисту 9.5 (фабрики вместо module-level глобалей делаются по ходу, в каждом сервисе сразу).

## 13. Бэклог (за рамками MVP, в порядке примерного приоритета)

1. Рефакторинг entrypoint'ов: избавиться от module-level глобалей (settings/broker/container/app) -> фабрики create_broker(settings)/create_app(), DI-контейнер собирается в main(); uvicorn factory-mode. Сначала новые сервисы, затем тем же паттерном payment_service/analytics_service.
2. Команда cancel_saga: отмена заказа пользователем во время активной саги (допустима только до pivot; после - отказ "уже оплачен").
3. notification_service (email-мок): notifications.commands + .dlq, retriable-шаги после pivot, компенсационные уведомления (9.12).
4. Параллельные шаги саги (fork/join) - вместе с notification_service.
5. Re-drive tooling: CLI или admin API мутации (retry шага, abort саги, переигровка из DLQ) с лимитом redrive_count.
6. Prometheus + Grafana в docker-compose; алерты (FAILED > 0, DLQ непустой, outbox stale, застрявшие саги).
7. Каталог как источник цены: validate_pricing (см. 12.2).
8. Confluent Schema Registry (или аналог) вместо статических JSON Schema - когда контрактов станет много.
9. Retention/архивация: processed_events, outbox(SUCCESS), терминальные саги, DLQ-топики - скрипты удаления по created_at или помесячное партиционирование (решение итерации 3: до тех пор не удаляем).
10. Трейсинг (OpenTelemetry): trace context в metadata сообщений, span на шаг саги.
